At the end of [the last post](/posts/development/building-secure-form-system-on-cloudflare/) - about the encrypted, pseudonymized interview system I built for my wife's pharmacy students - I promised two things: the real usage numbers from running it in production, and the mistakes I made so you can skip them. A full cohort has now used it to conduct and record patient interviews about herbal medicine. This is that follow-up.

The short version: the architecture held up. Two Workers, D1, KV, and R2, all on the free tier, carried a real workload without drama. The interesting parts were what running it in front of real users surfaced - one outright failure, a query I taught myself to stop running, and a privacy hole I'm glad I caught while it still mattered. All of it my doing; none of it Cloudflare's.

## 📊 What production actually looked like

Here is the cohort, by the numbers:

| Metric | Value |
|---|---|
| Students in the cohort | 130 |
| Peak concurrent students | 7 |
| Requests per active student | ~60 / minute |
| Shortest session | ~10 minutes |
| Median session | 55 minutes |
| Response writes on the busiest day | 300+ |

A few things stand out.

**The peak was small - and that's the point.** Seven students at once, out of a cohort of 130, is nothing by "scale" standards. No autoscaling, no load test, no war room. And yet seven concurrent users were enough to surface the bug below - because the problem was never load. It was the shape of a *single* request.

**Each active student was chatty.** Around 60 requests a minute per person. That's the autosave talking: the form persists as you type, so a focused student generates a steady stream of small writes. Multiply by seven and the busiest day crossed 300 response writes to D1 - still a rounding error against the free tier, but enough volume to make request shape matter.

**The ten-minute interview.** The shortest session lasted about ten minutes against a median of 55. You don't complete a 40-question patient interview in ten minutes by hand. The timing in the audit trail made it obvious that someone pasted in generated answers. The same timestamped log I built for GDPR compliance turned out to double as a behavioral signal - not something I'll act on, but a good reminder that *when* data arrives is itself data.

## 💥 The one thing that broke: too many open connections

The interview form has 40-plus questions. The first version of autosave did the obvious, naive thing: when the form saved, it persisted each answer as its own `INSERT` - one round trip to D1 per field - and it did this every couple of seconds as the student worked.

With one person testing, this is invisible. In production, with a save firing all of its inserts together, the backend Worker started returning errors about too many open connections.

Here are the numbers I should have known going in - including the one I got wrong. I'd half-remembered the simultaneous-connection limit as ten. It's six:

- A single Worker invocation can have at most **six connections open simultaneously** waiting for response headers. The [docs](https://developers.cloudflare.com/workers/platform/limits/) put it plainly: *"Each Worker invocation can have up to six connections simultaneously waiting for response headers."*
- On the **free plan**, one invocation can make at most **50 subrequests** in total - and every D1 query is a subrequest.

Firing one `INSERT` per answer for a 40-plus-field form walks straight into both ceilings inside a single request: far more than six writes in flight at once, and past 50 subrequests once you add the surrounding reads. The fix is to stop treating each answer as its own trip to the database.

<Mermaid chart={`graph LR
    subgraph before["Before: one INSERT per answer"]
        A1[Autosave] --> Q1[INSERT]
        A1 --> Q2[INSERT]
        A1 --> Q3["INSERT ×40+"]
        Q1 --> DB1[(D1)]
        Q2 --> DB1
        Q3 --> DB1
    end
    subgraph after["After: db.batch - one round trip"]
        A2[Autosave] --> BAT[db.batch] --> DB2[(D1)]
    end`} />

[Drizzle's](https://orm.drizzle.team/docs/batch-api) `db.batch()` collapses all of those statements into a single call to D1. The change was two lines:

```typescript
// src/routes/a/responses.ts - persist a student's answers
await Promise.all(                                    // [!code --]
  answers.map((a) => db.insert(responses).values(a)), // [!code --]
)                                                     // [!code --]
await db.batch(                                       // [!code ++]
  answers.map((a) => db.insert(responses).values(a)), // [!code ++]
)                                                     // [!code ++]
```

`db.batch()` sends every statement to D1 in one request and runs them inside an implicit transaction: *"If all statements succeed, the transaction commits. If any statement fails, the entire transaction rolls back and no changes are made."* That bought me two things at once. The obvious one: a 40-insert fan-out became a single round trip, comfortably under both the connection and subrequest ceilings. The one I didn't plan for: atomicity. D1 [doesn't support](https://developers.cloudflare.com/d1/worker-api/d1-database/#batch) a standard `BEGIN TRANSACTION`, so `db.batch()` is how you get all-or-nothing writes - a half-saved interview is now impossible.

<Alert type="info">
**The mental model that bit me:** on Workers, every `await db.something()` is a network hop, and there's a hard ceiling on how many can be in flight at once. Don't loop the database - batch it.
</Alert>

One caveat if you copy this pattern: D1 caps a single statement at 100 bound parameters, so a genuinely huge write needs chunking into several batched statements. For a 40-field form, you never come close.

## 🔍 Reading the query logs

The connection bug taught me to watch *how* each request talks to D1, so I started reviewing the query patterns regularly - which endpoints read the most, how many rows, how often. One path stood out by a mile: autosave.

Saving answers didn't just write - it first *read the entire form structure* to validate them. That read was a four-level nested query walking form → stages → questions, pulling back around 113 rows, and it ran on every single autosave. Across one 30-minute session that was roughly 120 reads of a form that hadn't changed once - about a third of all the time D1 spent working that session.

The form structure is the perfect thing to cache: an admin edits it occasionally, students read it for hours. So I moved it into KV. The assembled schema - only the validation-relevant bits (question id, type, config, required-or-not; no labels, no PII) - is cached under `form-cache:{formId}`, and autosave reads that instead of the database:

```typescript
// Autosave's hot read: KV first, D1 only on a miss
const cached = await kv.get<CachedFormData>(`form-cache:${formId}`, 'json')
if (cached) return cached                          // [!code highlight]

const data = await loadFormStructure(db, formId)   // one lightweight query
await kv.put(`form-cache:${formId}`, JSON.stringify(data), {
  expirationTtl: 86_400, // 24h
})
return data
```

The entry is invalidated the instant an admin edits the form, so students never see a stale structure. I worked the TTL up in stages as I trusted it more - 60 seconds, then 10 minutes, then 24 hours once invalidation was wired in. Net effect: that heavy structure query went from running on every keystroke to running a handful of times a day at most - once per form when someone first opens it, plus whenever an admin makes an edit. On a day nobody touches a given form, it doesn't run at all.

The most human thing I learned came from watching the admins, not the students. With results live in the dashboard, the admins checked the running average *every single day* the survey was open - and a number that moves daily is a number people start steering toward. So I put the results under embargo: until the survey closed, the dashboard still shows the anonymous comments (handy for catching the odd bug report) but hides every rating, distribution chart, and aggregate. It's a single cutoff datetime checked on each render; once it passes, the full view comes back on its own.

### The anonymity hole I almost shipped

The survey is anonymous by design, split across two tables: `satisfaction_participation` records *who* took part - so nobody is asked twice - and holds no answers, while `satisfaction_responses` holds the rating and comment with no link to a user. Clean separation. Except it wasn't.

Both rows are written in the same request handler, one right after the other, so a student's participation row and their anonymous response row were created in lockstep - and *insertion order* quietly tied them back together. An admin with raw SQL access could sort each table by `id`, zip row N to row N, and de-anonymize every comment. I only spotted it while the survey was already live.

The obvious fixes weren't enough, and that's the part worth remembering:

- Dropped the `created_at` columns from both tables. Order still leaked through the autoincrement `id`s.
- Dropped the `id` from the participation table and keyed it on `(user_id, form_id)` instead. Order *still* leaked - because SQLite gives every table a hidden `rowid`, and rows physically live in `rowid` (insertion) order whether or not you expose an id.

What finally closed it was recreating the table as [`WITHOUT ROWID`](https://www.sqlite.org/withoutrowid.html):

```sql
CREATE TABLE satisfaction_participation (
  user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  form_id INTEGER NOT NULL REFERENCES forms(id) ON DELETE CASCADE,
  status  TEXT NOT NULL CHECK(status IN ('completed', 'declined')),
  PRIMARY KEY (user_id, form_id)
) WITHOUT ROWID;
```

A `WITHOUT ROWID` table has no hidden rowid - it stores rows clustered by its composite primary key, here `(user_id, form_id)`, so the physical order reflects *who and which form*, never *when*. The insertion-order side channel is simply gone. (For good measure, the migration also shuffled the existing rows with `ORDER BY RANDOM()` as it copied them across.)

## 💰 The bill

All of this - two Workers, D1, KV sessions, R2 attachments, a whole cohort of students - ran inside Cloudflare's free tier. The 300-plus writes on the busiest day sit far below D1's free limits, and the geo-fencing WAF rule from the [previous post](/posts/development/building-secure-form-system-on-cloudflare/#-locking-the-front-door-waf-and-placement) keeps bot noise out, which keeps invocation counts honest. No AWS bill, no managed Postgres, no surprise at the end of the month.

## 💡 Closing thoughts

Production taught me the thing you can only learn by shipping: Cloudflare's primitives were never the risk. Workers, D1, KV, the service binding - all of it behaved exactly as designed. Every problem worth writing about came from my own data modeling: a write pattern that hit the connection limit, a read pattern that hammered the database on every keystroke, and a schema that leaked insertion order. The platform did its job; I had to learn to do mine.

Two lessons outlast this particular project. The limits that bite you on Workers aren't the ones on the pricing page - requests, storage, bandwidth - but the per-invocation ones: six open connections, fifty subrequests on the free plan. Shape each request to respect those and a free-tier Worker quietly carries a real workload. And the privacy lesson is even more durable: anonymity isn't a column you delete - it's the absence of every ordering an attacker can sort by, including the ones the database keeps for itself.
