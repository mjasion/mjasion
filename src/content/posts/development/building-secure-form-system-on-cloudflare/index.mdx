In a [previous post](/posts/development/why-i-chose-cloudflare-and-tanstack/), I explained why I chose Cloudflare and TanStack for my side projects. That post was about the stack. This one is about what happens when you actually build something with real security requirements on it.

My wife needed a system for her pharmacy students to conduct and record patient interviews about herbal medicine usage. I helped her build it. The data turned out to be sensitive - personal health information protected by GDPR (RODO in Poland) - so the system needed encryption at rest, pseudonymized identities, and audit trails. I built it entirely on Cloudflare Workers. This post describes the key architectural decisions and why I made them.

## 🏗️ Two Workers, one system

The architecture uses two Cloudflare Workers connected via a [service binding](https://developers.cloudflare.com/workers/runtime-apis/bindings/service-bindings/):

<Mermaid chart={`graph TD
    Browser -->|HTTPS/TLS 1.3| FW[Frontend Worker<br/>TanStack Start + React]
    FW -->|Service Binding<br/>env.API.fetch| BW[Backend Worker<br/>Hono API]
    BW --> D1[(D1 Database<br/>SQLite EU)]
    BW --> KV[(KV Store<br/>Sessions)]
    BW --> R2[(R2 Storage<br/>File attachments)]`} />

The frontend worker handles SSR with TanStack Start. The backend worker runs a [Hono](https://hono.dev/) API with all business logic, encryption, and database access. The service binding between them means the frontend calls the backend directly - no HTTP round-trip, no CORS, no public endpoint for the API.

In practice, the API call looks like this:

```typescript
// Frontend worker calls backend via service binding
const response = await env.API.fetch(url.toString(), init)
```

The hostname doesn't matter (`http://api.internal` is a convention). Cloudflare routes the call internally. The two workers share a cookie domain, so auth tokens flow between them seamlessly.

Each environment (local, staging, production) gets its own D1 database, R2 bucket, and KV namespace - all configured in `wrangler.jsonc`. No infrastructure provisioning, just config.

## 🗄️ Database design for encrypted data

D1 is the relational store - SQLite managed by Cloudflare in the EU region. I use [Drizzle ORM](https://orm.drizzle.team/) for type-safe queries and migrations.

The interesting design challenge is: **how do you query data that's encrypted?** You can't run `WHERE email = ?` against an AES-256-GCM ciphertext. The solution is a dual-key pattern:

```typescript
// Users table has two email columns:
email: text('email'),          // AES-256-GCM encrypted (for display)
emailHash: text('email_hash'), // SHA-256 hash (for lookup)
```

When a user logs in, I hash their email to find the row, then decrypt the stored email for display:

```typescript
export async function hashEmail(email: string, salt: string): Promise<string> {
  const normalizedEmail = email.toLowerCase().trim()
  const data = new TextEncoder().encode(normalizedEmail + salt)
  const hashBuffer = await crypto.subtle.digest('SHA-256', data)
  return bufferToHex(hashBuffer)
}
```

The hash is salted to prevent rainbow table attacks. The salt is stored as an environment secret - never in code.

**What's encrypted:**

| Data | Method | Why |
|------|--------|-----|
| Email, first name, last name | AES-256-GCM | PII under GDPR |
| Form response values | AES-256-GCM | Medical data |
| File attachment URLs | AES-256-GCM | Link to sensitive files |

**What's hashed:**
- Email (SHA-256) - for database lookup without decryption

**What's plaintext:**
- Form structure, question definitions, plant taxonomy - no PII

R2 handles file attachments (photos, documents) with Cloudflare's built-in encryption at rest. KV stores ephemeral data: sessions with 30-day TTL and magic link tokens with 15-minute TTL. Both auto-expire.

## 🔐 Authentication: two paths

The system serves two types of users with different auth flows.

### Magic links for administrators

Admins log in via email. The flow:

1. Admin enters email
2. Backend generates a one-time token (16 random bytes), stores it in KV with a 15-minute TTL
3. AWS SES sends the link
4. Admin clicks the link, token is verified and immediately deleted (one-time use)
5. JWT access token + KV refresh token are issued

The first user to log in is automatically promoted to admin. No seed data, no manual setup.

```typescript
const token = generateId() // 16 random bytes → hex string
await kv.put(`magic:${token}`, JSON.stringify({ email, ... }), {
  expirationTtl: ttl, // 15 minutes
})
```

### Plant-name credentials for students

Students don't use personal emails. Instead, admins generate credentials where usernames are Latin plant names - *Calendula officinalis*, *Hypericum perforatum*. The username itself carries no personal information. Students log in with their plant name and a password.

This was a deliberate goal: the system stores no student PII whatsoever - no name, no email, no student ID number. In the database, a student is nothing more than a plant name. Mapping a plant name back to a real person is impossible from the data alone; that link lives only outside the system, in whatever roster the admin keeps separately.

This is pseudonymization by design. Even if the database leaks, linking a response to a specific student requires knowledge that exists only outside the system.

### Session architecture

Both auth flows produce the same session structure:

| Token | Storage | Payload | Lifetime |
|-------|---------|---------|----------|
| Access (JWT) | HTTPOnly cookie | `userId` + `role` | 30 minutes |
| Refresh | HTTPOnly cookie + KV | `userId` + `role` | 30 days |

**Critical detail:** tokens contain only `userId` and `role`. Never a name, email, or any PII. If a token is intercepted, it reveals nothing about the person. The auth middleware auto-refreshes expired access tokens using the KV refresh token - transparent to the frontend.

Rate limiting runs on KV counters: 3 magic link requests per minute, 10 password attempts per minute per IP.

## 🔒 Three layers of encryption

Data is encrypted at three levels:

<Mermaid chart={`flowchart TB
    subgraph L1["Layer 1: Transport"]
        T1["TLS 1.3 - browser to Cloudflare edge"]
    end
    subgraph L2["Layer 2: Application"]
        T2["AES-256-GCM - PII and medical data"]
        T3["SHA-256 - email lookup hash"]
    end
    subgraph L3["Layer 3: Infrastructure"]
        T4["Encrypted at rest - D1 and R2 storage"]
    end
    L1 --> L2
    L2 --> L3`} />

The application-layer encryption uses the Web Crypto API - native to Workers, no npm dependencies:

```typescript
export async function encrypt(plaintext: string, keyBase64: string): Promise<string> {
  const key = await importKey(keyBase64)
  const iv = crypto.getRandomValues(new Uint8Array(12)) // Random IV per operation

  const ciphertext = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv },
    key,
    new TextEncoder().encode(plaintext)
  )

  // Output: base64(IV + ciphertext + authTag)
  const combined = new Uint8Array(12 + ciphertext.byteLength)
  combined.set(iv, 0)
  combined.set(new Uint8Array(ciphertext), 12)
  return bufferToBase64(combined)
}
```

Every encryption produces a different ciphertext (random IV), so identical inputs don't produce identical outputs. The encryption key is a 32-byte secret stored as a Cloudflare environment variable - accessible only at runtime, never logged, never committed.

## 📋 Audit trail

Every significant action is logged to an `auditLogs` table: form creation, update, deletion, login, logout, response submission, admin actions - over 35 action types in total. Each log entry stores a JSON diff of old and new values, enabling both compliance auditing and debugging.

This isn't just for security theater. When a student reports a lost response or an admin needs to verify who accessed what, the audit trail provides a complete, queryable history.

## 💡 Closing thoughts

The entire system - two Workers, D1 database, R2 storage, KV sessions - runs within Cloudflare's free tier for the current usage levels. No AWS account, no managed Postgres.

If you're building something that handles sensitive data, you don't need a dedicated security team or enterprise infrastructure. You need good encryption primitives, sensible architecture, and a platform that gives you the building blocks. Cloudflare Workers - with Web Crypto, D1, and KV - turned out to be exactly that.
